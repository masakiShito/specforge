# Quality Principles

SpecForge は「書けること」ではなく「正しく保てること」を重視する。

## 1. 品質定義をコード化する
- 品質はレビュー時の暗黙知にしない
- `packages/lint-rules` にルールとして実装可能な形へ落とし込む
- 「指摘できる」ではなく「自動検出できる」を目標にする

## 2. 設計書は構造化データとして保存する
- 自由文主体の運用に戻さない
- `packages/document-schema` で document / section / field / reference を明示する
- 将来的な互換性のため、スキーマにはバージョン概念を導入する

## 3. 正規化を前提にした編集体験を作る
- 入力順や表記揺れは `packages/editor-engine` で正規化する
- UI は記述補助を担うが、正しさの最終判定は engine + rules が担う

## 4. 出力より内部整合性を優先する
- Markdown / HTML / PDF は派生物であり、真実のソースは内部モデル
- 出力フォーマット都合でコアモデルを汚染しない

## 5. 再現性を最優先する
- 誰が書いても最低品質を下回らない状態を目指す
- レビュー観点は再利用可能な規則として蓄積する
