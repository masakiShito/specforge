# SpecForge Vision

## 背景
多くの設計書は Markdown やドキュメントテンプレートから始まるが、実際には執筆者ごとの習慣に強く依存し、品質と粒度がばらつきやすい。
その結果、レビュー負荷の増大、実装解釈のズレ、再利用困難な知識が組織内に蓄積される。

## 課題
1. 設計書の品質が執筆者スキルに依存する
2. 設計内容の構造が明示されず、機械的な検査が困難
3. Markdown 中心の運用では、整形規則と設計規則が混在しやすい
4. 「書けたかどうか」は判断できても、「正しいかどうか」を担保しづらい

## 解決したいこと
SpecForge は、設計書を自由文ではなく**構造化データ**として扱う。

- 設計対象を section / field / reference / table などの意味単位で定義する
- UI は執筆体験を担うが、設計規則は document-schema と lint-rules に集約する
- editor-engine は Markdown 非依存の内部モデル（AST・正規化・変換）を維持する
- 品質ルールにより、必須項目・整合性・参照性を継続的に検査する

## 目指す世界観
- 誰が書いても、最低品質を下回らない設計書が生成される
- 設計書がレビュー対象であると同時に、再利用可能な知識資産になる
- 出力形式（Markdown / HTML / PDF）が変わっても、内部の正しさが先に保証される
- 「書きやすさ」より「崩れにくさ」を優先し、組織的な再現性を高める
