## アプリ概要

SpecForgeは「スキーマ駆動の構造化設計書エディタ」。事前定義されたスキーマ（Document → Section → Field）に従い、フォーム形式で設計書を記述する。自由記述ではなくスキーマ制約によって設計書の品質と一貫性を担保する。

## 画面構成

3ペイン構成（CSS Grid: `240px 1fr 320px`）。

| ペイン | 幅 | 役割 |
|--------|-----|------|
| 左 | 240px | セクション一覧・選択 |
| 中央 | 残り全幅 | 選択セクションのフィールド入力フォーム |
| 右 | 320px | バリデーション結果・ドキュメントJSONプレビュー |

## 主要コンポーネント

- `DocumentEditor` — メインコンテナ。状態管理・バリデーション計算・3ペインレイアウト
- `SectionList` — セクション選択UI。未入力必須項目の件数バッジ表示
- `SectionForm` — 選択セクションのフィールドをフォームとして描画
- `FieldRenderer` — valueTypeに応じた入力コンポーネントの出し分け
- `ValidationPanel` — 必須項目の未入力警告一覧 + JSON出力

## データ構造

`Project > Document > Section > Field` の4階層。Fieldにはスキーマ情報（label, required, valueType, options等）が定義され、ユーザー入力値は `Record<string, FieldValue>` で別管理される。

## 現在できること

- サンプルドキュメント（Checkout Screen）の閲覧・編集
- セクション間の切り替え
- text / textarea / boolean / enum 型フィールドへの入力
- 必須項目の未入力チェック（リアルタイム）
- 入力内容のJSON形式プレビュー
- セクション別の未入力件数バッジ表示

## 現在できないこと

- number型フィールドの入力（UIプレースホルダ表示のみ）
- table型フィールドの入力（UIプレースホルダ表示のみ）
- reference型フィールドの入力（UIプレースホルダ表示のみ）
- ドキュメントの新規作成・切り替え
- データの保存・永続化（ブラウザ上のメモリ内のみ）
- スキーマの動的編集
- ドキュメント間のリファレンス解決
- フィールド値のフォーマット検証（数値範囲、文字数制限等）
- 複数プロジェクトの管理
- ER仕様・ビジネスルール仕様のプリセット

## 制限事項

- バリデーションは「必須項目の未入力検出」のみ

# SpecForge 使い方ガイド

## 1. このツールについて

### 何をするツールか

SpecForgeは、ソフトウェア設計書を構造化された形式で作成するエディタです。

通常のMarkdownエディタやテキストエディタとは異なり、**事前に定義されたスキーマ（型定義）に従って、フォーム形式で設計書を記述します**。

### なぜこの構造なのか

設計書の品質問題の多くは「書き手によるばらつき」に起因します。

- 書くべき項目が抜ける
- 書き方がチームメンバーごとに異なる
- レビュー時に構造の不備を指摘しなければならない

SpecForgeは、スキーマによって「何を書くべきか」「どの項目が必須か」を事前に定義し、入力フォームとして提供することで、これらの問題を解消します。

### Markdownとの違い

| 観点 | Markdown | SpecForge |
|------|----------|-----------|
| 構造 | 書き手が自由に決定 | スキーマで事前定義 |
| 項目の網羅性 | 書き手に依存 | 必須項目として強制 |
| 入力形式 | 自由テキスト | フィールド型に応じたフォーム |
| 品質のばらつき | 大きい | スキーマにより最小化 |
| 出力 | Markdown文書 | 構造化JSON |

## 2. 画面の見方

画面は3つのペインで構成されています。

### 左ペイン：セクション一覧（幅240px）

ドキュメントに含まれるセクションの一覧を表示します。

- セクション名をクリックすると、中央ペインにそのセクションの入力フォームが表示されます
- 選択中のセクションは青枠でハイライトされます
- 未入力の必須項目がある場合、セクション名の右側に赤い数字バッジが表示されます（例：`2` なら未入力必須項目が2件）
- 左ペイン上部には、ドキュメントのタイトル・種別・バージョンが表示されます

### 中央ペイン：編集エリア（残り全幅）

選択中のセクションのフィールドを、フォーム形式で表示します。

- セクションヘッダーに、セクション名と「必須セクション」「任意セクション」の区分が表示されます
- 各フィールドにはラベルが表示されます
- 必須フィールドにはラベル横に赤い「必須」バッジが付きます
- フィールドの入力形式は、スキーマで定義された `valueType` に基づいて自動で決まります

### 右ペイン：バリデーション・JSONプレビュー（幅320px）

2つのセクションで構成されています。

**バリデーション：** 必須項目の入力状態を表示します。

- 未入力の必須項目がある場合：赤字で「セクション名 — フィールド名: Required field is empty」と一覧表示
- すべての必須項目が入力済みの場合：緑字で「未入力の必須項目はありません」と表示

**ドキュメント JSON：** 現在の入力内容をJSON形式でリアルタイム表示します。

- ドキュメントのID、キー、タイトル、種別、バージョン、各セクション・フィールドの入力値が含まれます
- 暗色背景のコードブロックとして表示されます

## 3. 基本操作

### セクションを選択する

左ペインのセクション名をクリックします。選択したセクションの入力フォームが中央ペインに表示されます。

アプリ起動時は、最初のセクションが自動的に選択されます。

### フィールドを入力する

中央ペインに表示された各フィールドに値を入力します。フィールドの型（valueType）に応じて、入力方法が異なります。

| 型 | 入力方法 |
|----|----------|
| text | テキスト入力欄（1行） |
| textarea | テキスト入力欄（複数行、リサイズ可） |
| boolean | ドロップダウン（「未選択」「はい」「いいえ」） |
| enum | ドロップダウン（スキーマ定義の選択肢から選択） |

入力した値はリアルタイムで状態に反映されます。

### バリデーションを確認する

右ペイン上部のバリデーション欄を確認します。必須項目が未入力の場合、該当するセクション名とフィールド名が赤字で表示されます。

左ペインのセクション名横の赤いバッジでも、セクションごとの未入力件数を把握できます。

### 状態を確認する

右ペイン下部のドキュメントJSONを確認します。各フィールドの `value` プロパティに現在の入力値が反映されています。未入力のフィールドは `value` が未定義の状態で表示されます。

## 4. 用語説明

| 用語 | 意味 |
|------|------|
| **Document** | 1つの設計書。種別（kind）とバージョンを持つ。複数のSectionで構成される |
| **Section** | 設計書内の論理的なまとまり。「概要」「画面項目」など。複数のFieldで構成される |
| **Field** | 1つの入力項目。ラベル・入力型・必須/任意などのスキーマ情報を持つ |
| **key** | Document / Section / Field に付与される識別用の文字列。プログラム上の一意識別に使用 |
| **required** | その項目が必須かどうかを示すフラグ。trueの場合、未入力はバリデーション警告の対象となる |
| **valueType** | フィールドの入力形式を決定する型指定。text / textarea / boolean / enum / number / table / reference の7種類が定義されている |
| **validation** | 入力内容の検証。現在は必須項目の未入力チェックのみ実装されている |

## 5. 入力形式（valueType）

### 対応済み

| valueType | 入力UI | 説明 |
|-----------|--------|------|
| **text** | テキスト入力欄（1行） | 短いテキストを入力する場合に使用。プレースホルダ：「テキストを入力」 |
| **textarea** | テキスト入力欄（複数行） | 長い説明文を入力する場合に使用。縦方向にリサイズ可能。最小高さ100px |
| **boolean** | ドロップダウン（3択） | はい/いいえの二択。選択肢：「未選択」「はい」「いいえ」。未選択状態ではundefinedとして扱われる |
| **enum** | ドロップダウン（定義済み選択肢） | スキーマで事前定義された選択肢から1つを選択。初期状態は「選択してください」 |

### 未対応

| valueType | 現状 | 備考 |
|-----------|------|------|
| **number** | 未対応 | UIに「この型（number）は現在未対応です」と表示される |
| **table** | 未対応 | UIに「この型（table）は現在未対応です」と表示される。スキーマ上はcolumns定義・minRows/maxRows指定が可能 |
| **reference** | 未対応 | UIに「この型（reference）は現在未対応です」と表示される。スキーマ上はdocument/section/field/externalの4種類のリファレンスが定義されている |

## 6. バリデーションの考え方

### なぜバリデーションがあるのか

設計書の品質を担保するためです。

自由記述では「書くべきことを書き忘れる」問題が発生します。SpecForgeでは、スキーマで必須と定義された項目が未入力の場合、即座に警告を表示します。これにより、レビュー前の段階で項目の抜け漏れを防止します。

### 必須項目の意味

スキーマ定義で `required: true` が設定されたフィールドは、値の入力が必要です。

以下の場合に「未入力」と判定されます：

- 値が未定義（一度も入力されていない）
- text / textarea / enum 型で値が空文字列

boolean型は「未選択」状態（undefined）の場合に未入力と判定されます。false（「いいえ」）は有効な入力値として扱われます。

### 警告の扱い

現在のバリデーションは**警告（Warning）**として表示されます。入力をブロックする機能はありません。すべてのフィールドに自由に値を入力でき、警告は参考情報として提示されます。

## 7. 制限事項

### 未対応機能

- **number / table / reference 型の入力**：スキーマ定義は存在するが、入力UIは未実装
- **ドキュメントの新規作成**：現在はサンプルドキュメント1件のみ閲覧・編集可能
- **ドキュメントの切り替え**：複数ドキュメントの選択・切り替え機能は未実装
- **データの保存・永続化**：入力内容はブラウザのメモリ上にのみ存在し、リロードで消失する
- **エクスポート**：JSONの表示はあるがファイル出力機能は未実装
- **ER仕様・ビジネスルール仕様のプリセット**：DocumentKindとして定義されているが、プリセットは未作成

### 注意点

- ブラウザをリロードすると、すべての入力内容が初期状態に戻ります
- サンプルデータにはtable型フィールドが含まれていますが、入力はできません
- バリデーションは必須項目の未入力チェックのみです。文字数制限・数値範囲・形式チェック等は行われません

### 現在の仕様制約

- スキーマ（ドキュメント構造）は実行時に変更できません。これは設計上の意図的な制約です
- フィールド値はフラット構造で管理されるため、ネストしたデータ（表の行・列など）は現状扱えません
- UIの言語は日本語固定です

## 8. このツールの設計思想

### 一貫性

誰が設計書を書いても、同じ構造・同じ項目・同じ形式になることを目指しています。スキーマが構造を規定するため、書き手の裁量による構造のばらつきが発生しません。

### 再現性

スキーマ定義を共有すれば、同じ種類の設計書を何度でも同じ品質で作成できます。テンプレートとは異なり、スキーマは入力形式と必須項目を強制するため、「テンプレートを無視して自由に書く」問題が起きません。

### 品質担保

必須項目のバリデーションにより、最低限書くべきことが書かれていない状態を検出できます。自由記述では見逃しがちな項目の抜け漏れを、ツールレベルで防止します。

## 改善提案（コードから読み取れる範囲）

### UX改善

- **フィールド説明の表示**：Field型には `description` と `placeholder` プロパティが定義されているが、`description` はUIに表示されていない。フィールドの入力意図をユーザーに伝えるため、ラベル下に説明文を表示すべき
- **入力済み/未入力の視覚的区別**：現在、未入力フィールドと入力済みフィールドのスタイルに差がない。入力済みフィールドにチェックマーク等を付与すると進捗把握が容易になる
- **セクションの説明表示**：Section型には `description` プロパティがあるが、UIに表示されていない
- **キーボードナビゲーション**：セクション間をキーボードで移動する手段がない

### 構造改善

- **FieldValueの型安全性**：`FieldValue = string | number | boolean | undefined` で全型を共用しているが、valueTypeに応じた型ガードを入れると安全性が向上する
- **バリデーション結果のメッセージ国際化**：バリデーション警告メッセージが英語（"Required field is empty"）だが、UIの他の部分は日本語。統一が必要
- **状態管理の永続化準備**：`DocumentEditorState` がReactのuseState内にのみ存在する。localStorage等への永続化層を追加する準備として、状態のシリアライズ/デシリアライズを分離すべき

### バリデーション改善

- **enum値の選択肢検証**：現在、enum型は空文字列チェックのみ。スキーマ定義の `options` に含まれる値かどうかの検証を追加すべき
- **カスタムバリデーションルール**：Field型にバリデーションルール（最大文字数、正規表現パターン等）を定義できるプロパティを追加すべき
- **セクション単位の完了状態**：セクション内の全必須フィールドが入力済みかどうかを明示的に判定し、セクション一覧に完了マークを表示すべき
- **バリデーションの重大度レベル**：現在は全てWarningだが、Error（入力不可値）とWarning（推奨）を区別できると実用性が向上する
